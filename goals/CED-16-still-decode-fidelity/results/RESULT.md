# CED-16 Result: Fix Still Decode Serving Embedded Previews

## What changed

Commits (on `CED-16-still-decode-fidelity`, stacked on CED-15's locked
head `90ec3fe`):

- `6f59216` fix: decode stills full-size, not embedded previews
- `fc61f1d` fix: gate full-res still decode on page landing
  (wave-001 claude-code #1)

The one-option bug from the validated repro: `StillDecoder.decode`
passed `kCGImageSourceCreateThumbnailFromImageIfAbsent`, which serves
the container's embedded preview whenever one exists — and
iPhone-camera HEICs always embed one, so a 6000×4000 source decoded to
its 432×288 embedded thumbnail (~90× pixel deficit) instead of the
4096-long-edge viewer ceiling.

`StillDecoder` now routes per source family
(`App/MobileSeal/Detail/StillDecoder.swift`):

- **Normal images** decode with
  `kCGImageSourceCreateThumbnailFromImageAlways` at the existing 4096
  ceiling — regenerated from the full-size image, never the embedded
  preview.
- **RAW/DNG** keeps the embedded-preview path (`IfAbsent`) — the
  distinction the shipped comment intended but the option didn't
  deliver (CED-12 wave-001 coderabbit #5). Detection
  (`usesEmbeddedPreview`): source UTI conforming to
  `public.camera-raw-image`, falling back to the per-index RAW/DNG
  properties dictionaries for generically-sniffed containers. Note
  the real ImageIO key is `kCGImagePropertyRawDictionary` — the
  `kCGImageSourcePropertyRawDictionary` spelling in GOAL.md's scope
  prose does not exist (wave-001 coderabbit #1; spec text left as
  promoted, code uses the real key).

The routing seam (`decodeOptions(for:)` + `usesEmbeddedPreview(_:)`)
is internal-but-exposed so the regression tests assert the option
choice directly — ImageIO refuses to DECODE a synthetic camera-less
DNG (verified empirically on macOS and the simulator), so pixels
cannot prove the RAW path in a hermetic test; classification +
option routing can be and are.

The reconciliation fix (`fc61f1d`): with stills genuinely decoding at
the ceiling (~44 MiB RGBA at 4096 px), the eager decode in
`viewDidLoad` would have let 2–3 preloaded pager neighbors hold
full-res bitmaps simultaneously — the pre-fix 432-px previews masked
this overshoot of CED-11's one-still-plus-headroom budget. The full
decode now starts in `didLand()` (all three landing paths route
through the pager's `landed(on:)`); neighbors keep their 512-px
poster, swipe-away cancels, re-landing retries only if a full decode
never succeeded.

Fixtures (committed, reproducible via the new
`Scripts/generate-still-fixtures.swift`, same committed-batch
convention as `generate-fixtures.swift`):

- `App/Fixtures/still-embedded-6000x4000.heic` — the validated repro

  bug, baked into the fixture so it can never silently stop
  reproducing).
- `App/Fixtures/still-embedded-1200x800.heic` — sub-ceiling image

  generator asserts the classification. ImageIO's RAW pipeline

## What did NOT need changing

- **Cache purge on upgrade (scope item 2)**: verified, not assumed —
  there is no persisted decoded-still cache to purge. The full-screen
  decode lands only in the page's `imageView.image`
  (`MediaPageViewController.decodeFullStill`), session-scoped by
  construction; no `NSCache`/disk cache of decoded stills exists in
  any app target (swept App/ + Sources/). The only persisted derived
  artifact is the 512-px encrypted thumbnail store, generated by
  `Thumbnailer` with `FromImageAlways` — never affected by this bug.
  Previously-viewed items therefore re-decode sharp on next view with
  no migration step. (Both interactive reviewers independently
  reached the same conclusion.)
- `Thumbnailer` — already `Always`; its comment correctly deferred
  the preview-reuse distinction to the detail viewer.
