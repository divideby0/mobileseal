# Codex blind plan review — Streaming Media Playback

## Blocking concerns

1. **`ChunkSource` conflates two incompatible abstractions.** The plan describes a public random-access playback/fetch interface, but VaultCore already defines `ChunkSource` as a move-only, sequential plaintext import source with `read(into:)` and `rewind()` for two-pass hashing ([GOAL.md, Workstream A](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:26); [Gallery.swift, existing ChunkSource](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/Sources/VaultCore/Gallery.swift:4)). Reusing or publishing it would mix plaintext ingestion, sealed-CAS retrieval, and authenticated playback; the plan must specify separate interfaces, such as sealed-chunk fetch by `ChunkAddress` and custody-bound plaintext range reading.

2. **The resource-loader protocol is not sufficiently planned to produce a correct implementation.** Workstream B mentions only byte ranges and `isByteRangeAccessSupported`, omitting `contentType`, `contentLength`, `currentOffset`, `requestsAllDataToEndOfResource`, incremental responses, retained request lifetimes, cancellation, and exactly-once success/error completion ([GOAL.md, Workstream B](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:42)). Apple requires accepted requests to be fully satisfied or failed and requires all-to-end requests to continue until EOF, cancellation, or error; without an explicit request state machine, this can hang playback or truncate the perceived asset ([content-information contract](https://developer.apple.com/documentation/avfoundation/avassetresourceloadingcontentinformationrequest), [data-request contract](https://developer.apple.com/documentation/avfoundation/avassetresourceloadingdatarequest)).

3. **The plan incorrectly treats a time seek as an app-controlled byte-range mapping.** `AVPlayer` parses the container and decides which potentially overlapping or out-of-order byte ranges to request; the delegate does not receive a time and independently choose one chunk ([GOAL.md, Workstream B.2](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:49)). Byte-exact MOV/MP4 originals may have their `moov` sample tables near the tail, so “prefetch leading chunks” cannot guarantee autoplay or play-while-downloading; the plan needs leading- and trailing-`moov` fixtures and must follow actual loader requests rather than assume linear access ([GOAL.md, Workstream C.1](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:61); [Apple network-optimized movie behavior](https://developer.apple.com/documentation/avfoundation/avassetwriter/shouldoptimizefornetworkuse)).

4. **The resident-plaintext budget has no custody-compatible ownership model.** Existing reads produce move-only `SecureBytes` exposed only through scoped borrowing, while `readRange` allocates both an exact-length result and a chunk-sized scratch buffer ([ChunkReader.swift, range reading](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/Sources/VaultCore/ChunkReader.swift:63); [SecureBytes.swift, custody contract](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/Sources/VaultCore/SecureBytes.swift:42)). An LRU must define noncopyable cache entries, pinned borrowers, concurrent misses, eviction while borrowed, zeroization, and oversize-request behavior; furthermore, `respond(with: Data)` necessarily creates ordinary-heap plaintext outside that cache and AVFoundation’s buffers cannot be counted or evicted by VaultCore.

5. **Playback is not integrated into the existing sole-owner lock protocol.** The coordinator currently owns the move-only session, drops gallery/index state, clears only the thumbnail reader, and then invokes the custodian’s bounded drain ([VaultCoordinator.swift, lock](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/App/MobileSeal/VaultCoordinator.swift:407); [KeyCustodian.swift, drain](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/Sources/VaultCore/KeyCustodian.swift:134)). The plan never identifies who retains each player, delegate, reader, cache, and outstanding loading request, nor the lock ordering needed to cancel and fail those requests before draining; “AVPlayer buffers purge” is not an enforceable API guarantee ([GOAL.md, Workstream C.3](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:73)).

6. **The current app cannot import the ordinary video used by the gates.** `PHPickerConfiguration` excludes videos, `StagedPart` has no ordinary-video role, `ImportEngine` requires a `.still`, and `Thumbnailer` is ImageIO-only ([PickerMediaProvider.swift, picker filter](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/App/MobileSeal/Import/PickerMediaProvider.swift:102); [MediaProvider.swift, part roles](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/App/MobileSeal/Import/MediaProvider.swift:21); [ImportEngine.swift, still requirement](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/App/MobileSeal/Import/ImportEngine.swift:194)). The plan’s “poster frame and duration” line omits the required provider, import-role, AVAsset thumbnailing, metadata/index, recovery, and UI model changes, making Green Gate 2 impossible as written ([GOAL.md, Green Gate 2](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:97)).

7. **The external-playback and capture policies collide at the proposed API boundary.** The plan enables external playback while blanking whenever `UIScreen.isCaptured` is true, but Apple defines that state as including recording, mirroring, and AirPlay, so a blanket shield can blank the very external playback the settled decision requires ([GOAL.md, Workstream C.3](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:73); [Apple `isCaptured`](https://developer.apple.com/documentation/uikit/uiscreen/iscaptured)). `UIScreen.isCaptured` is also deprecated on the latest-OS target; the plan needs an explicit truth table using current scene capture state plus `AVPlayer.isExternalPlaybackActive`, and a HITL AirPlay/external-display gate rather than merely assigning a default-true property.

8. **The chunk-profile benchmark cannot support its promised decision.** It lacks a committed container/codec/resolution/GOP/`moov` matrix, warm-versus-cold cache controls, first-presented-frame instrumentation, repetitions, percentile statistic, thermal/storage controls, and a predeclared 4 MiB-versus-1/2 MiB decision threshold ([GOAL.md, Workstream D](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:82)). Seek completion is not first-frame presentation, and reimporting identical plaintext into one gallery will trigger existing dedup and preserve the first entry’s `chunkSize`, so variants require separate vaults or a controlled fixture generator ([Gallery.swift, dedup profile reuse](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/Sources/VaultCore/Gallery.swift:186)). “Thresholds stated in-test” permits post-hoc acceptance and a device “spot-check” does not make simulator-derived latency suitable for a storage-format policy.

9. **Several green gates assert properties for which no observable is defined.** “Frames render,” “buffers purge,” “no plaintext temp file,” and “prefetch stays within budget” need precise probes, failure conditions, and observation boundaries ([GOAL.md, Green gates](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:93)). The existing custody helper only scans files beneath a supplied filesystem root and cannot inspect AVFoundation’s process-internal buffers or private storage ([TestSupport.swift, canary scanner](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/App/MobileSealTests/TestSupport.swift:70)); the plan needs request/range/decrypt/cache counters, active-request counts after lock, and a first-new-pixel-buffer timestamp.

## Advisories

1. **The ticket combines too many independent risk surfaces.** It simultaneously redesigns VaultCore reads, anticipates remote fetching, implements AVFoundation playback, expands import and metadata, builds three pager transitions, adds Live Photo motion, replaces still decoding, and runs a format-policy benchmark ([MAP.md, Streaming Media Playback ticket](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/wayfinder/MAP.md:33)). Streaming still decode and remote-source proof are separable scope leaks that can obscure whether the playback custody path itself is correct.

2. **The fake remote source has no availability or integrity semantics.** “Slow/partial availability” does not specify whether a missing sealed chunk suspends, retries, reports unavailable, supports cancellation, verifies the CAS address before decryption, or distinguishes permanent corruption from not-yet-downloaded data ([GOAL.md, Workstream A.3](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:38)). Those distinctions determine whether resource requests can be finished correctly and whether lock can promptly cancel a pending fetch.

3. **Autoplay prefetch has no request-priority or cancellation policy.** Neighbor preparation can cause multiple `AVPlayerItem`s to request headers, tail indexes, and media simultaneously, while hard-snap paging can invalidate that work before it completes ([GOAL.md, Workstream C.1–C.2](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:61)). A cache byte limit alone does not bound decrypt work, I/O, battery cost, ordinary `Data` copies, or late callbacks; the plan needs one-active-player rules and generation-token cancellation analogous to the existing thumbnail purge discipline.

4. **Memory-pressure behavior is not measurable as specified.** No initial cap, minimum cap, shrink ratio, hysteresis, recovery behavior, or rule for “all entries pinned while over budget” is stated ([GOAL.md, Workstream A.2 and D.2](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:33)). Deterministic tests need an injected pressure/available-memory provider; a live OS notification or `os_proc_available_memory` snapshot cannot reliably prove the claimed degradation path.

5. **“Streaming still decode” lacks a format-specific success definition.** ImageIO incremental input is generally sequential and may still require most or all source bytes before producing a full-resolution image, especially for HEIC/RAW containers, so this work cannot promise video-like random range efficiency ([GOAL.md, Workstream B.4](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:55); [StillDecoder.swift, current bounded decode](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/App/MobileSeal/Detail/StillDecoder.swift:5)). Its gate should specify peak ordinary-memory reduction and supported JPEG/HEIC/ProRAW behavior rather than merely replacing whole-file materialization.

6. **Unsupported-video behavior is absent.** “Video-only” does not define the accepted MOV/MP4 codecs, HDR/Dolby Vision/ProRes cases, malformed container handling, or whether unsupported-but-authentic media is “damaged,” “no preview,” or “not playable” ([GOAL.md, Workstream C.3](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:73)). The fixture suite needs at least supported MOV and MP4 cases plus a valid unsupported-codec case so loader failure is not misreported as AEAD damage.

7. **The reference list omits the implementation-defining material.** It cites results and a research report but not Apple’s resource-loading request/content-information/cancellation documentation, AirPlay/capture behavior, or the local files containing the existing `ChunkSource`, picker filter, metadata model, coordinator lock path, and canary limitations ([GOAL.md, References](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:110)). The map also says to extend playback vocabulary in `CONTEXT.md`, but no work item or gate requires that update ([MAP.md, Notes](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/wayfinder/MAP.md:19)).

## Questions the plan leaves unanswered

1. **What exactly crosses the proposed source seam?** Is it an authenticated sealed chunk addressed by `ChunkAddress`, an unverified stored chunk object, or plaintext bytes, and which type is responsible for CAS-address verification versus AEAD verification ([GOAL.md, Workstream A.1](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/goals/drafts/20260720-011003-streaming-media-playback/GOAL.md:28))?

2. **Who owns playback custody and in what order is it revoked?** The plan does not name the object that retains the reader, loader delegate, accepted requests, decrypt tasks, cache, player item, and observers, or how that owner participates in the coordinator’s single lock path.

3. **What does the residency budget actually count?** It is unclear whether the cap covers only cache-owned `SecureBytes`, active decrypt scratch, response `Data`, loader-request payloads, decoded frames, posters, and neighbor players, and what happens when pinned memory alone exceeds the cap.

4. **How are all-to-end and unavailable-range requests handled?** The plan does not say whether the delegate incrementally feeds such requests, delays them pending remote chunks, imposes a maximum response slice, or how cancellation unblocks a suspended source without decrypting the remainder ([Apple `requestsAllDataToEndOfResource`](https://developer.apple.com/documentation/avfoundation/avassetresourceloadingdatarequest/requestsalldatatoendofresource)).

5. **What exact evidence selects a chunk profile?** The plan leaves unanswered which percentile and threshold decide among 4, 2, and 1 MiB, whether the result must hold cold on the physical iPhone, and whether startup, sequential playback, scrub latency, memory, and object-count regressions are jointly considered.

6. **How are external playback and capture distinguished?** The settled decisions require both, but the plan does not state whether AirPlay external playback is exempted from the capture shield, whether generic screen mirroring remains blanked, or how those branches are tested on real hardware.

7. **What metadata evolution supports videos and existing Live Photos?** `MediaMetadata` has no duration or ordinary-video kind, while `MediaItem` has no paired-video length or duration ([MediaMetadata.swift, schema v1](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/App/MobileSeal/MediaMetadata.swift:15); [MediaIndex.swift, MediaItem](/Users/openclaw/src/divideby0/mobileseal/.worktrees/drafts/20260720-011003-streaming-media-playback/App/MobileSeal/MediaIndex.swift:10)). The plan does not choose a backward-compatible optional-field extension, a schema-version bump, or the recovery behavior for already-imported paired videos.
