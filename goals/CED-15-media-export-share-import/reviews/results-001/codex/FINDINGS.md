## 1. Export cancellation is starved by the export actor

Evidence: `ExportController.stage` creates an unstructured `Task` at `App/MobileSeal/Export/ExportController.swift:85` from within an actor-isolated method. That task inherits the actor executor and then performs the entire synchronous `for item in plan` / `decryptToFile` loop without an `await` (`:94-105`, `:189-207`). While it is reading and writing a large video, `prepareForLock` / `cancelActiveExport` cannot enter the actor to execute `stagingTask?.cancel()` at `:169-174`; cancellation can occur only after staging has already finished. The cancellation test at `App/MobileSealTests/ExportCustodyTests.swift:191-197` does not establish that staging has begun before racing cancellation, and it calls a final `prepareForLock`, so it can pass whether cancellation wins before startup or staging runs to completion. In addition, the `.grace`/`.off` background path merely launches `Task { await export.cancelActiveExport() }` at `App/MobileSeal/VaultStore.swift:460-463`, unlike the lock path's `beginBackgroundTask`, so iOS may suspend the process before a queued teardown completes.

Why it matters: the goal specifically requires lock/background to cancel in-flight decrypt/writes, await handles, and sweep plaintext. With a large export, lock can remain stuck behind a full decrypt, and under grace/off backgrounding the app can be suspended with a partially or fully decrypted export still in `StagingExport/` until relaunch. This is both a custody-contract failure and an unbounded lock-latency regression.

Suggested fix: run the blocking decrypt/write work on an explicitly nonisolated worker (for example a `Task.detached` created from Sendable plan/reader/container values, or an async I/O worker that does not inherit `ExportController` isolation), while retaining the task handle in the actor so teardown can cancel it immediately and await completion. Begin an iOS background-task assertion synchronously before scheduling the grace/off cancellation and end it only after `cancelActiveExport()` finishes. Strengthen the test with a blocking/chunk-reader seam that signals after the first write, then request lock/background and assert cancellation reaches the reader before it is allowed to finish.

## 2. Quota enforcement is racy and can evict data for an item that never commits

Evidence: `InboxStore.enforceQuota` scans the directory, computes totals, and deletes committed items at `App/ShareInbox/InboxStore.swift:254-285`; only afterwards does `InboxWriter` write the new manifest at `App/ShareInbox/InboxWriter.swift:146-159`. There is no file lock, `NSFileCoordinator`, SQLite transaction, or other synchronization shared by extension processes. Two writers can therefore both scan below a one-item/2-GiB boundary, both pass, and both publish manifests, exceeding the advertised hard quota. Independently, a single writer can delete one or more old committed items and then fail or be terminated before line 158, permanently losing the old items even though the incoming item never committed. The concurrent-writer test at `App/MobileSealTests/InboxProtocolTests.swift:409-429` checks UUID filename collision only; it does not place writers at a quota boundary or inject failure after eviction.

Why it matters: concurrent invocations are explicitly in scope, and the 2-GiB/50-item limit is the mechanism preventing an unlock-free extension from consuming unbounded shared-container storage. The current read/modify/delete sequence can violate that bound and can destructively expire user data for a failed share.

Suggested fix: place quota calculation, incoming commit publication, notice update, and evictions behind a real interprocess transaction. A small SQLite metadata database is the clearest option; alternatively use an advisory lock plus a durable journal/recovery protocol, with the incoming payload already complete before taking the lock. Do not irreversibly delete prior committed items until the new commit is durably represented. Add deterministic two-writer boundary tests and an injected manifest-write/termination failure after quota planning.

## 3. Extension cancellation does not cancel or await the writer

Evidence: `ProviderInboxAttachment.loadFileRepresentation` discards the `Progress` returned by `NSItemProvider` at `App/ShareInbox/InboxWriter.swift:35`. `loadCopy` uses `withCheckedThrowingContinuation` at `:255-278`, which has no cancellation handler, so canceling the Swift task neither cancels the provider request nor resumes the continuation. `ShareViewController.cancel` then calls `stagingTask?.cancel()` and immediately calls `extensionContext?.cancelRequest` at `App/MobileSealShare/ShareViewController.swift:115-121`; it does not await the writer. The unit test at `App/MobileSealTests/InboxProtocolTests.swift:387-404` awaits `task.value` for roughly the fake provider's full delay, which is not what the production Cancel button does.

Why it matters: the extension process may be terminated as soon as the request is canceled, before `stageOne` reaches its cleanup catches. A cancel during copy/hash can strand fresh incomplete plaintext until the 15-minute stale sweep, and a cancel after the last cancellation check can still publish a manifest, making an item the user canceled appear in the app. This contradicts the goal's typed cancellation-and-cleanup contract.

Suggested fix: make the attachment seam return a cancelable handle, bridge it with `withTaskCancellationHandler`, and ensure a continuation is resumed exactly once on cancellation/provider completion. In the view controller, disable the UI, cancel the task, await its cleanup result, and only then call `cancelRequest`. Add a controller-level test that exercises the actual Cancel action and asserts the extension context is completed only after the inbox contains neither committed nor incomplete files.

## 4. Batch claiming is not atomic on disk

Evidence: `InboxStore.claim(itemIDs:galleryID:)` writes one `.claim.json` per item in a plain loop at `App/ShareInbox/InboxStore.swift:213-223`. If the second or later atomic write fails (for example under low disk), earlier claim markers remain. `VaultStore.acceptInboxImport` returns `false` from the switchboard body on that error and clears `activeInboxClaim` at `App/MobileSeal/VaultStore.swift:330-342`, but it never releases the markers already written. Subsequent scans classify those items as claimed and omit them from prompts/settings; recovery occurs only if the process is relaunched and `sweepAtLaunch` releases them.

Why it matters: accepting a prompt is specified as one atomically gallery-bound batch. A recoverable I/O error instead splits the batch and makes valid staged items disappear for the rest of the session.

Suggested fix: represent the whole batch with one atomically renamed claim record containing the gallery and item IDs, or implement rollback of every marker written before an error (plus launch recovery for a crash during rollback). Keep `activeInboxClaim` until rollback has completed, and add an injected-failure test on the Nth claim write.

## 5. Oldest-first expiry is not totally ordered

Evidence: manifests encode `committedAt` using `JSONEncoder.DateEncodingStrategy.iso8601` at `App/ShareInbox/InboxManifest.swift:111-115`. The tests themselves note this representation's second granularity (`App/MobileSealTests/InboxProtocolTests.swift:243`). `InboxStore.scan` sorts only on the decoded `committedAt` at `App/ShareInbox/InboxStore.swift:165`, while manifests first pass through a dictionary. Attachments staged serially in a normal batch commonly commit within the same second, so after a rescan their dates tie and the item selected by `removeFirst()` for quota expiry is filesystem/dictionary-order dependent rather than the oldest.

Why it matters: quota expiry deletes user-provided staged media. Even when deletion is expected, choosing an arbitrary member rather than the promised oldest one makes retention behavior nondeterministic and can discard a newer item first.

Suggested fix: persist fractional-second dates (custom ISO-8601 formatting) and add a stable collision-resistant tie-breaker, such as a monotonic sequence allocated inside the same quota transaction or `(committedAt, itemID)` with a documented ordering. Test multiple commits within one second and assert a deterministic oldest-first eviction.
