# Report

## Executive Summary

**Best default for the main app.** For a 0.5–1s interactive gallery unlock on current iPhones/iPads/Vision Pro, the best evidence-backed default is libsodium Argon2id with `opslimit=3` and `memlimit=268,435,456` bytes, i.e. 256 MiB. This is libsodium’s MODERATE class: its documentation says INTERACTIVE is a 64 MiB baseline, MODERATE is 256 MiB and about 0.7s on a 2.8 GHz Core i7, and SENSITIVE is 1 GiB and about 3.5s in the [pwhash API](https://doc.libsodium.org/password_hashing/default_phf). Since libsodium’s Argon2id wrapper uses one lane internally, the directly relevant knobs are memory and passes rather than an exposed parallelism setting; for a local unlock where only one derivation runs at a time, spending 256 MiB in the foreground app is a reasonable modern Apple-device target.

**Lower and upper bounds.** The absolute floor should be above OWASP’s minimum Argon2id profiles, not at them: OWASP lists equivalent minimum settings such as 19 MiB with 2 iterations or 46 MiB with 1 iteration and says password hashing should generally take under one second in its [Password Storage](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html) cheat sheet. RFC 9106 is more aggressive: it recommends Argon2id at 2 GiB, t=1, p=4 when memory is available, and 64 MiB, t=3, p=4 for memory-constrained environments in its [Argon2 profile](https://www.ietf.org/rfc/rfc9106.html). Libsodium cannot expose p=4 through `crypto_pwhash`, so the closest libsodium-friendly mapping is 64 MiB/`opslimit=3` as a conservative extension or low-end fallback, 256 MiB/`opslimit=3` as the main default, and 512 MiB/`opslimit=3` or `opslimit=4` only after per-device timing confirms it stays under the unlock budget.

**Benchmarks are suggestive, not definitive.** Public mobile-specific Argon2id benchmarks for A17/A18/Vision Pro are sparse; the most useful published numbers are adjacent evidence. A SwiftArgon2 README reports on an Apple M2 Max that libsodium with p=1 completes its default 64 MiB, 3-pass profile in about 0.081s and notes that libsodium uses hand-tuned NEON for BLAKE2b in its [Apple Silicon benchmark](https://github.com/mimiclone/argon2-swift/blob/main/README.md). A separate .NET/libsodium benchmark shows near-linear scaling, with 256 MiB/t=3 at about 331–392 ms, 512 MiB/t=3 at about 689–815 ms, and 1 GiB/t=3 at about 1.4–1.7s on desktop/laptop hardware in its [Argon2 results](https://github.com/samuel-lucas6/benchmark-argon2-dotnet). Those figures support using 256 MiB as the safest default and treating 512 MiB as an opt-in calibrated tier rather than a universal setting.

**Comparable apps cluster below or beside this.** Bitwarden defaults to Argon2id at 32 MiB, 6 iterations, and 4 threads, and warns iOS users about high Argon2 memory in extension contexts in its [KDF settings](https://bitwarden.com/help/kdf-algorithms/). Standard Notes uses Argon2id with 64 MiB, 5 iterations, and p=1 for its protocol 004 root key in its [encryption whitepaper](https://standardnotes.com/help/security/encryption). Cryptomator iOS is not an Argon2 comparator because it derives its vault KEK with non-parallel scrypt and stores the scrypt parameters in the vault metadata according to its [security architecture](https://docs.cryptomator.org/security/architecture/). 1Password is also not an Argon2 comparator today: it uses PBKDF2-HMAC-SHA256 with 650,000 iterations plus its Secret Key design, as described in its [PBKDF2 support](https://support.1password.com/pbkdf2/) page.

**iOS caveats matter as much as timing.** Apple’s memory model can terminate processes that exceed a resident memory limit, and Apple explicitly notes that extensions have much lower limits than foreground apps in its [jetsam reports](https://developer.apple.com/documentation/xcode/identifying-high-memory-use-with-jetsam-event-reports) documentation. Apple also exposes thermal states and says higher thermal states can require apps to reduce CPU, GPU, and I/O usage in its [thermal guidance](https://developer.apple.com/library/archive/documentation/Performance/Conceptual/power_efficiency_guidelines_osx/RespondToThermalStateChanges.html). Therefore, a production gallery app should calibrate on first launch or first password set, cache only the unwrapped vault key under biometric/device protection after a successful password unlock, and reserve the full Argon2 cost for cold password entry rather than every gallery open.

## Conclusion

**Recommended starting point.** For a libsodium `crypto_pwhash(..., ALG_ARGON2ID13)` gallery unlock on A17/A18 and M‑series iPhone/iPad/Vision Pro hardware, start at `opslimit=3` and `memlimit=256 MiB`—libsodium’s MODERATE preset—and keep an adaptive calibration step that raises memory toward 384–512 MiB only if the measured cold unlock remains below about 0.75s on the specific device and thermal state. This recommendation is grounded in libsodium’s own statement that MODERATE uses 256 MiB and takes about 0.7s on a 2.8 GHz Core i7, while SENSITIVE uses 1 GiB and about 3.5s; newer Apple mobile SoCs should generally make 256 MiB feasible for a main-app unlock, but the exact number must be measured because Argon2 throughput is memory-bandwidth- and implementation-dependent in the [libsodium pwhash](https://doc.libsodium.org/password_hashing/default_phf) guidance.

**Practical ceiling.** Do not use `crypto_pwhash_MEMLIMIT_SENSITIVE` as the default for an interactive gallery unlock: it is 1 GiB, is intended by libsodium for highly sensitive non-interactive work, and is likely to be brittle under iOS memory pressure, multitasking, Low Power Mode, and thermal throttling. If the unlock may ever run inside an iOS extension such as Photos/Share/AutoFill-style flows, use a separate extension-safe profile around `opslimit=4–6` with `memlimit=32–64 MiB`, because KeePassium documents an approximately 120 MB total AutoFill memory allowance and recommends reducing Argon2 memory to about 32 MB for extension compatibility in its [AutoFill memory](https://support.keepassium.com/kb/autofill-memory/) guidance. Store the chosen parameters with the wrapped gallery key and rehash/rewrap upward later when calibration shows stable headroom.
